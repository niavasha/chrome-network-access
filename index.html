Browser Permissions & Network Access Probe

A local page to see what data modern browser APIs expose

Understand your browser. Make informed permission choices.

Modern browsers expose a rich set of APIs — geolocation, camera, Bluetooth, USB, and the new Chrome Local Network Access prompt — and each click of "Allow" shares something specific with the site. Most of the time the site has a legitimate need; sometimes it doesn't. Either way, the choice is easier when you know exactly what the data looks like.

This is a transparency tool: it runs each API against your own browser, shows you the real output, and explains what it reveals. Use it to sanity-check your defaults, test how your Chrome responds to the new LNA prompt, and see whether your browser is correctly blocking the classic silent leaks (WebRTC, fingerprinting entropy).

The good news: modern Chrome defends you in depth. Even when you grant Local Network Access, sites still can't read responses from devices that don't opt in with CORS headers. Real local IPs are hidden behind mDNS hostnames by default. The prompts exist so you can make an informed choice — and the choice usually leaks less than you'd fear.

🔒 Privacy: nothing is collected, stored, or transmitted.
This page runs entirely in your browser. No telemetry, no analytics, no cookies, no remote fetches to any server we control — all probe results stay on your device and vanish when you close the tab. There is no backend. You can verify by reading the source or opening DevTools → Network and watching what requests leave (only the probes you deliberately click). If you hit Download JSON the file saves locally; it is never uploaded.

Open in Chrome (or any Chromium browser). Click each button to trigger that API. Results and plain-English interpretation appear in each card. Raw JSON is collapsible underneath.

Local Network Probe silent/LNA

Attempt to fetch common private addresses. In Chrome 130+ served from a public origin, this triggers the Local Network Access prompt.

Why this permission exists (and why it's a good thing)
Before LNA, any website you visited could silently send HTTP requests to your router (192.168.1.1), printers, NAS, smart TVs, and localhost — no prompt, no log, nothing. Attackers abused this for years:
  • Router CSRF — malicious ads on any site could POST to your router's admin panel and change your DNS, redirecting all future traffic through an attacker.
  • IoT exploitation — drive-by attacks against unpatched cameras, printers, and smart-home hubs reachable only on your LAN.
  • Silent LAN reconnaissance — fingerprinting your network shape for ad-tech and fraud scoring.
LNA closes these attacks. Now a public site has to ask first, and even after you allow, the target device must opt in with a CORS header to expose its responses. That's defence in depth: permission + same-origin policy + target opt-in. This prompt showing up on shopping sites means Chrome is doing its job — surfacing something that used to happen invisibly so you can make an informed call.
Probe results will be interpreted here: live device / absent / refused.

Localhost Port Scan silent in stable Chrome

Probe common ports on 127.0.0.1. Timing of fetch failures reveals which ports are open, closed, or filtered. In current default Chrome, this happens without any prompt regardless of origin.

Why would a website scan your ports? (real, not theoretical)
Fair question. Here's what's actually been found in the wild:
  • eBay (2020) — scanned 14 ports including 5900 (VNC), 3389 (RDP), and remote-access tool ports to fingerprint visitors and detect bots. Documented by Dan Nemec.
  • LexisNexis / ThreatMetrix — bank and retail anti-fraud vendor, scans localhost to detect whether you're running known RAT (Remote Access Tool) ports, which correlates with account-takeover attempts.
  • Bot-detection SDKs (PerimeterX, DataDome, Kount) — check for Chrome's remote-debugging port 9222 as a signal of automation/scraping.
  • Developer-machine fingerprinting — open 5432 (Postgres), 27017 (Mongo), 11434 (Ollama), 3000 (Node dev) strongly implies "this is a developer" — a different risk profile for fraud scoring and a different ad audience.
  • DNS-rebinding exploits — once a site maps your open ports, it can use DNS rebinding to actually talk to those services (unauthenticated Jupyter notebooks, dev databases, router admin panels).
Status today: still silent. The Local Network Access specification covers loopback (127.0.0.1), but Chrome deliberately deferred loopback enforcement in the 130 rollout — too many legitimate developer, Electron, and local-agent workflows depend on localhost access. In default stable Chrome, these probes fire with no prompt. Loopback gating is available behind chrome://flags/#local-network-access-checks and via enterprise policy, but neither is on by default. This card reveals what your machine currently exposes — which is why it matters.
Whatever ports are open here are what any website you visit can currently map.
